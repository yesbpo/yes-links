## What was built

Two additive, non-breaking props on `YesLinksDashboard` (and its sub-components) for multi-tenant embedding:

### `clientScope?: { campaign?: string; tags?: string[] }`
Scopes all data-fetching to a specific client's subset. Every `GET /links` and `GET /dashboard/summary` request automatically appends:
- `?campaign=<value>` when `clientScope.campaign` is set
- `?tags[]=<v1>&tags[]=<v2>` (repeated params, no double-encoding) when `clientScope.tags` is set
- No scope params when `clientScope` is absent — backward-compatible with existing integrations

### `mode?: 'internal' | 'external'` (default `'internal'`)
Controls what an end-user sees in the Create Link overlay:
- `'internal'` — full form: campaign + tags editable (developer/admin use)
- `'external'` — campaign + tags fields hidden from the UI; values are silently applied from `clientScope` (`lockedCampaign`, `lockedTags`) so the end-user cannot inspect or modify them

## Technical decision: `buildUrl` returns `URL`, not `string`

Attempting to pass `'tags[]'` as a key to `URLSearchParams.set()` encodes the brackets as `%5B%5D`, and the subsequent `.append('tags%5B%5D', v)` call causes a second encoding: `tags%5B%5D%5B%5D=value`. 

Fix: `buildUrl` returns a `URL` object. Callers append array params with `url.searchParams.append('tags[]', v)` directly — the API object handles encoding exactly once. Callers call `.toString()` when they need a string.
